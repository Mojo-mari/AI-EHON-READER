# えいご絵本リーダー 方針書

最終更新：2026-03-27（本文テキストの保存先議論を追記）

## 1. プロダクト概要

絵本をカメラで撮影するだけで、AIがえいごを読み取り・読み上げてくれるWebアプリ（PWA）。
対象ユーザーは英語絵本を子どもに読み聞かせたい保護者・子ども本人。

---

## 2. マネタイズ方針

### 基本モデル：フリーミアム × 月額サブスクリプション

APIコスト（OCR・TTS・翻訳）が使うたびに発生するため、買い切りではなくサブスクが適している。

### プラン設計

| プラン | 内容 | 価格 |
|--------|------|------|
| 無料 | 月5回まで読み取りOK・本棚保存なし | 0円 |
| プレミアム | 読み取り無制限・本棚保存無制限 | 月額300〜500円（要最終決定） |

### 決済方針

- **Stripe**を使いWeb経由で決済する
- App Store内課金（手数料30%）は使わない
- PWA経由でインストールしてもらい、決済はWebブラウザ上で完結させる

---

## 3. 配布方針（PWA）

### 採用：PWA化してApp Store / Google Playにも登録

| 方法 | 評価 |
|------|------|
| Webアプリのまま | 手軽だが競合に発見性で負けやすい |
| 今すぐネイティブアプリ化 | 工数が大きすぎる |
| **PWA化（採用）** | **Webのコードを流用しつつApp Storeにも登録できる** |
| 将来のネイティブ化 | ユーザーが大幅に増えた段階で検討 |

PWA（Progressive Web App）とは：Webアプリをアプリっぽく動かす仕組み。ホーム画面への追加・フルスクリーン表示が可能で、Google PlayおよびApp Storeへの登録もできる。

### PWA対応状況

- manifest.ts：設定済み
- アイコン（192px / 512px）：生成済み
- App Store申請：未実施

## 4. データ保存のライン（法的リスク対策）

⚠️ **本文テキストの保存先は現在議論中（2026-03-27時点）。以下の比較を参照して後日最終決定すること。**

現在の実装：**ローカル保存（localStorage）**を採用中。

---

### 【検討中】本文テキスト：サーバー保存 vs ローカル保存

#### 議論の背景

ログイン機能の実装にあたり、「機種変してもデータが引き継げるか」という問題が浮上した。
本文テキストをサーバーに保存しないと、ログインしても新しい端末では本棚の中身が復元されず、既存ユーザーの離脱につながる可能性がある。

#### 比較表

|  | ローカル保存（localStorage） | サーバー保存（Supabase） |
|--|---------------------------|------------------------|
| 機種変時のデータ引き継ぎ | できない | できる |
| 著作権リスク（運営側） | ほぼゼロ | 低い（条件付き） |
| 著作権リスク（ユーザー側） | ユーザー次第 | ユーザー次第（変わらない） |
| 著作権者から発見されるリスク | ほぼゼロ | 存在する（低いが0ではない） |
| 実装の複雑さ | シンプル | やや複雑 |
| 既存ユーザーの離脱リスク | 高い（機種変で消える） | 低い |

#### ローカル保存のメリット・デメリット

メリット：
- 私的複製（著作権法第30条）がほぼ確実に適用される
- 第三者の設備を経由しないため著作権者から発見されない
- 実装がシンプル

デメリット：
- 機種変・ブラウザキャッシュ削除でデータが消える
- ログインを実装しても本棚の引き継ぎができない
- 既存ユーザーの離脱につながる可能性

#### サーバー保存のメリット・デメリット

メリット：
- 機種変してもデータが完全に復元できる
- ユーザー体験が大幅に向上する
- DropboxやGoogle Driveと同等のサービスとして成立する

デメリット：
- 私的複製（著作権法第30条）の適用が「条件付き」になる（グレーゾーン）
- 著作権者からの削除申請・訴訟リスクが生まれる（低いが0ではない）
- まねきTV事件（最高裁2011年）の判例で、1対1通信でも「自動公衆送信装置」とみなされるリスクが示されている

#### サーバー保存する場合のリスク軽減条件

以下をすべて満たせば、DropboxやGoogle Driveと同等のリスク水準に収まると考えられる：

- RLS（行レベルセキュリティ）でユーザーごとに完全分離し、他人のデータは絶対に見えない
- 共有機能を実装しない
- 削除申請に対応できる窓口を設置する（contact@eigoehonjp.com）
- 利用規約に「著作権を有するコンテンツのみ使用可」と明記する

#### 現時点の方針（暫定）

ログイン機能の実装（フェーズ1後半）に合わせて**サーバー保存を採用**し、上記のリスク軽減条件をすべて満たす実装を行う。ただし後日改めて議論し、ローカル保存への変更も技術的には可能（`src/lib/bookshelf.ts` の変更のみで対応できる設計にしてある）。

## 6. リリースロードマップ

### フェーズ1：無料リリース（現在）

- ログイン機能なし
- 本棚データはローカル保存（ブラウザ内）のみ
- 完全無料でユーザーを集める
- PWA化してホーム画面追加・App Store登録を目指す
- APIコストはユーザーが少ない間は許容範囲内

### フェーズ2：サブスク導入

移行タイミングの判断基準（案）：月間アクティブユーザーが一定数を超えたとき、またはAPIコストが月○円を超えたとき（要最終決定）

- ログイン機能を追加（課金状態の管理のみに使用）
- 無料プランに読み取り回数の上限を設ける（月5回）
- プレミアムプランで制限解除（Stripe経由で決済）
- 既存ユーザーへは移行期間（数ヶ月）を設けて事前告知する
- フェーズ1の早期ユーザーへの優遇措置を検討（永久無料・割引など）

### フェーズ3：ネイティブアプリ化（要検討）

- ユーザーが大幅に増えた段階で検討
- React Native等での作り直しが必要
- PWAとネイティブの二重管理コストに注意
